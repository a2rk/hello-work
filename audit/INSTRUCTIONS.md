# Как пользоваться этим набором

> TL;DR: один файл-промпт скармливаешь Claude каждую итерацию, он сам выберет следующий таск и сделает один шаг.

---

## Что лежит в `audit/`

| Файл | Что внутри | Кто его трогает |
|---|---|---|
| `CONTEXT.md` | Описание продукта, архитектуры, правил выполнения, и **task ledger** со статусами `[ ] / [~] / [x]` | Claude правит статусы тасков; ты — только если хочешь добавить таск или поправить приоритет |
| `DRIVER.md` | **Промпт**, который ты копируешь в Claude каждую итерацию | Никто не трогает после создания |
| `INSTRUCTIONS.md` | Этот файл — для тебя | Никто не трогает |

---

## Workflow

### Каждая итерация

1. Открой Claude Code в этой репе.
2. Скопируй ВЕСЬ текст из `audit/DRIVER.md` (от строки «Ты — senior macOS engineer» до конца) и вставь в чат.
3. Жди отчёт. В отчёте Claude скажет: ID таска, что сделал, был ли релиз, какой следующий.
4. Проверь визуально что не накосячил (`git diff`, `git log`, посмотри файлы которые он правил).
5. Если всё ОК — переходи к следующей итерации. Если что-то поплыло — попроси откатить (`git restore .`) и переделать.

### Нюансы

- **Claude НЕ накапливает контекст между сессиями** — он каждый раз заново читает `CONTEXT.md`. Поэтому ledger со статусами ОБЯЗАТЕЛЬНО должен быть актуальным после каждой итерации; Claude обновит его сам.
- **Каждый второй таск — verify** (по дизайну). Не пропускай — это страховка от регрессий.
- **Релизы происходят пакетами** — после закрытия фазы целиком. Например, после Phase A (тасков 1-10) поднимется версия (0.9.22 → 0.9.23) с подытогом фазы.
- **Если Claude нашёл что-то новое во время verify** — он создаст follow-up таск в конце ledger'а. Это нормально, ledger живой.

## Если что-то идёт не так

### «Claude взял не тот таск»

Скажи ему: «Ты должен был взять TASK-NNN, не TASK-MMM. Откати свои правки через git restore и попробуй ещё раз». Потом перезапусти промпт.

### «Регрессия в собранном бинаре»

Включи dev mode (10 тапов по «Данные»), посмотри Diagnostics. Скинь содержимое в чат Claude'у с просьбой создать follow-up таск или починить.

### «Хочу добавить свой таск»

Допиши руками в `audit/CONTEXT.md` в нужную фазу или в секцию `## 5. Follow-up tasks`. Формат как у остальных — `[ ] **TASK-NNN [impl]**` + следующая строка `[ ] **TASK-NNN+1 [verify]**`. Claude дойдёт до него по очереди.

### «Хочу пропустить таск»

Помеси его `[x]` руками с комментарием в скобках, например `[x] **TASK-NNN [impl]** ~skipped: not relevant after refactor~`. Claude увидит и не возьмёт.

### «Claude закрыл фазу но не релизнул»

Скажи: «Фаза X закрылась, нужен релиз. Прогони bump → build → package → commit → tag → gh release create как описано в DRIVER.md шаг 3a пункт 9».

### «Хочу запустить новый полный аудит»

Когда все таски `[x]`, попроси: «Запусти 4 параллельных Explore-агента по всему коду как в первый раз и сгенери новый набор тасков в `audit/CONTEXT.md`». Это создаст phase O с новыми находками.

---

## Что НЕ делать

- ❌ Не запускай несколько Claude-сессий параллельно над этим ledger'ом — они затрут правки друг друга в `CONTEXT.md`.
- ❌ Не редактируй файлы продакта (`Sources/...`) пока итерация в работе — Claude может перетереть.
- ❌ Не делай `git reset --hard` если итерация в `[~]` — потеряешь прогресс.
- ❌ Не правь `audit/DRIVER.md` без необходимости — это сильно поменяет поведение всех будущих итераций.

---

## Ожидаемая динамика

- **Тасков всего**: 60 (30 impl + 30 verify), плюс возможные follow-up'ы.
- **Фаз**: 14 (A–N).
- **Темп**: 1 итерация = 1 таск. Реалистично 5-15 итераций в день.
- **Срок**: 4-10 дней при регулярном прогоне.
- **Промежуточные релизы**: ~14 (по одному на фазу), все patch-версии в линейке 0.9.22 → 0.9.35.
- **Финальный релиз**: 0.10.0 (минорный bump, signal «major polish pass»).

## Если хочешь автоматизировать дальше

Можно завернуть в shell-скрипт, который:
1. Запускает Claude Code с `-p "$(cat audit/DRIVER.md)"`
2. Ждёт пока он закончит
3. Делает `git log` чтобы убедиться что коммит появился (или нет)
4. Повторяет

Но я бы НЕ автоматизировал полностью — глазами проходить итерации полезно, ловит косяки до коммита. Лучше так: один день руками — научишься понимать ритм; потом, если уверен, — крутишь в фоне.

## Финальная проверка перед стартом

1. У тебя в репо чисто (`git status` зелёный)?
2. Текущая ветка `main`?
3. self-signing cert «HelloWork Self-Signed» в keychain (`security find-identity -p codesigning ~/Library/Keychains/login.keychain-db | grep "HelloWork Self-Signed"`)?
4. Можешь собрать вручную (`swift build`)?

Если все четыре да — можно стартовать. Поехали.
