# Как пользоваться этим набором

> TL;DR: один файл-промпт скармливаешь Claude каждую итерацию, он сам выберет следующий таск и сделает один шаг.

---

## Что лежит в `audit/`

| Файл | Что внутри | Когда трогать |
|---|---|---|
| `CONTEXT.md` | Цель цикла, архитектурный контекст, правила, **task ledger**, метрики, follow-ups | **Под каждый цикл** перепиши секции 1, 2, 5, 6. Секции 3, 4 стабильны. Claude правит статусы тасков по ходу |
| `DRIVER.md` | **Промпт**, который ты копируешь в Claude каждую итерацию | Не трогаем без необходимости — изменения влияют на поведение всех итераций |
| `INSTRUCTIONS.md` | Этот файл — для тебя | Не трогаем |

---

## Запуск нового цикла

1. Перепиши `CONTEXT.md` секции:
   - **1. Goal** — один абзац: что за цикл, зачем сейчас.
   - **2. Architecture context** — какие файлы трогаем, инварианты для текущей задачи, что из соседнего НЕ трогаем (на случай если scope выходит за стандартный).
   - **5. Task Ledger** — фазы A..N с тасками. Каждый impl-таск = 1 строка с описанием + Файлы + Acceptance. Каждый verify-таск = 1 строка с указанием на impl выше.
   - **6. Metrics «9.5/10»** — конкретные проверяемые критерии успеха.
2. Стартуй первый таск через DRIVER.md prompt (см. ниже).

---

## Workflow итерации

1. Открой Claude Code в этой репе.
2. Скопируй ВЕСЬ текст из `audit/DRIVER.md` (от строки «Ты — senior macOS engineer» до конца) и вставь в чат.
3. Жди отчёт. Claude скажет: ID таска, что сделал, версия релиза, какой следующий.
4. Проверь визуально (`git diff`, `git log`, посмотри файлы которые он правил, mount свежий DMG если build pipeline касался).
5. Если всё ОК — следующая итерация. Если поплыло — `git restore .` + перезапуск промпта.

### Через `/loop`

Если хочешь батчем — запускай `/loop делай дальше, не ограничивайся одной таской`. Claude в dynamic-mode сам себе ставит `ScheduleWakeup` и работает блоками пока не остановишь.

### Нюансы

- **Claude НЕ накапливает контекст между сессиями** — он каждый раз заново читает `CONTEXT.md`. Поэтому ledger со статусами ОБЯЗАТЕЛЬНО должен быть актуальным после каждой итерации; Claude обновит его сам.
- **Каждый второй таск — verify** (по дизайну). Не пропускай — это страховка от регрессий.
- **Релизы — каждый таск**, не пакетами. Это даёт granular rollback и видимый прогресс через GitHub Releases.
- **Если Claude нашёл что-то новое во время verify** — он создаст follow-up в секции 7. Ledger живой.

## Если что-то идёт не так

### «Claude взял не тот таск»

«Ты должен был взять TASK-XNN, не TASK-YMM. Откати правки через `git restore .` и попробуй ещё раз».

### «Регрессия в собранном бинаре»

Включи dev mode (10 тапов по «Данные»), посмотри Diagnostics. Скинь содержимое в чат.

### «Хочу добавить свой таск»

Допиши руками в `audit/CONTEXT.md` в нужную фазу или в секцию `## 7. Follow-up tasks`. Формат как у остальных. Claude дойдёт по очереди.

### «Хочу пропустить таск»

Помечай `[x] **TASK-XNN [...]** ~skipped: <причина>~`. Claude увидит как done и не возьмёт.

### «Claude закрыл цикл, нужен ещё»

Когда все таски `[x]`, попроси: «Запусти audit по [модуль], ищи слабые места, сгенери Phase X с новыми тасками». Это создаст продолжение в том же файле.

---

## Что НЕ делать

- ❌ Не запускай несколько Claude-сессий параллельно над этим ledger'ом — затрут друг друга.
- ❌ Не редактируй файлы продакта (`Sources/...`) пока итерация в работе.
- ❌ Не делай `git reset --hard` если итерация в `[~]` — потеряешь прогресс.
- ❌ Не правь `audit/DRIVER.md` без необходимости.
- ❌ Не меняй bundle ID, не пересоздавай self-signed cert — это сломает TCC у юзеров.

---

## Финальная проверка перед стартом цикла

1. У тебя в репо чисто (`git status` зелёный)?
2. Текущая ветка `main`, тег с прошлого релиза?
3. Self-signing cert «HelloWork Self-Signed» в keychain (`security find-identity -p codesigning ~/Library/Keychains/login.keychain-db | grep "HelloWork Self-Signed"`)?
4. Можешь собрать вручную (`swift build` → 0 errors)?
5. CONTEXT.md секции 1, 2, 5, 6 переписаны под новый цикл?

Если все пять — да, можно стартовать.
