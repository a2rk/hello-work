# HelloWork Audit Driver

> Этот файл — **промпт для Claude**. Закидывается каждую итерацию пока в `audit/CONTEXT.md` есть хоть один `[ ]` таск. Каждая итерация двигает ledger ровно на один таск (либо impl, либо verify).

## Промпт целиком (копируй с этой строки и ниже)

Ты — senior macOS engineer + product/PM. Работаешь над проектом HelloWork (`/Users/igor/Code/Swift/FocusNap/HelloWork`). Цель сессии — выполнить ровно ОДИН следующий таск из `audit/CONTEXT.md`, аккуратно и без сюрпризов для соседних флоу.

### Шаг 1 — Контекст

1. Прочитай `audit/CONTEXT.md` ЦЕЛИКОМ. Это твой единственный источник истины: что мы строим, какая архитектура, какие правила выполнения тасков, текущий ledger со статусами.
2. Пробеги глазами по последнему commit'у в репо (`git log --oneline -5`) — пойми, что было сделано в прошлой итерации.
3. Если в проекте есть незафиксированные правки (`git status`) — это значит прошлая итерация не довела дело. Прочитай diff (`git diff`), оцени надо ли продолжать с того места или откатить и начать заново. Сообщи юзеру.

### Шаг 2 — Выбор таска

Читай ledger в `CONTEXT.md` сверху вниз. Возьми **самый первый таск со статусом `[ ]`**.

- Если таск помечен `[impl]` → перейди к Шагу 3a.
- Если таск помечен `[verify]` → перейди к Шагу 3b.

Помни жёсткое правило: verify-таск нельзя делать раньше связанного с ним impl-таска. Если ledger корректен — это и так невозможно (impl всегда стоит выше verify в списке).

### Шаг 3a — Implementation

1. **Пометь таск `[~]` (in_progress)** в `CONTEXT.md` сразу.
2. Открой все файлы из секции «Файл(ы)» таска и прилежащие.
3. **Подумай**: «Зачем эта функция / поле / свойство существует? Что должно происходить идеально?» Если задача вступает в конфликт с инвариантами из секции «Ключевые подсистемы и их инварианты» — стоп, обсуди с юзером.
4. Реализуй минимально-достаточное изменение, следуя «Глобальным правилам выполнения тасков» из CONTEXT.
5. Если на пути требуется ввести новые i18n строки — обязательно en + ru + zh, проверь порядок полей в Translation.swift (он строгий).
6. Если правка задевает hider/permissions — добавь точки `devlog(...)` для дальнейшей диагностики.
7. **Соберись**: `swift build 2>&1 | grep -E "error:" | head -10`. Если ошибки — фикси пока чисто.
8. **Пометь таск `[x]`** в `CONTEXT.md`.
9. **Релиз (всегда, для каждого таска)**:
    - `./scripts/bump.sh patch`
    - `./scripts/build.sh && ./scripts/package.sh && ./scripts/build_stub.sh && ./scripts/package_stub.sh`
    - в `dev_log.json` добавить entry: customMessage (1 предложение про этот таск), 2-4 points
    - `git add -A && git commit -m "Hello work X.Y.Z — TASK-NNN: <короткое название>"`
    - `git tag vX.Y.Z && git push origin main && git push origin vX.Y.Z`
    - `gh release create vX.Y.Z dist/HelloWork-X.Y.Z.dmg dist/HelloWork.dmg --title "..." --notes "..."`
10. Финально — отчёт юзеру в чат: что сделал, какой таск закрыт, какая версия релиза, какой следующий таск.

### Шаг 3b — Verification

> ⚠️ Verify-таск НЕ редактирует код продакта. Только читает, тестит, документирует.

1. **Пометь таск `[~]` (in_progress)** в `CONTEXT.md`.
2. Найди связанный impl-таск (на одну строку выше в ledger'е). Перечитай его описание + acceptance.
3. Прочитай diff impl-таска: `git log --grep="TASK-NNN" --all` или последние коммиты, найди те что относятся.
4. Прогони verification по чеклисту:
   - Соответствует ли изменение описанию проблемы и acceptance criteria?
   - Не сломаны ли соседние пути (см. инварианты из CONTEXT)?
   - Если impl касался hider/permissions — собрать `swift build`, проверить что compile-чисто.
   - Smoke-сценарии из таска (если описаны).
5. **Если всё OK** — пометь таск `[x]`. **Релиз тоже делай** (per-task политика, см. шаг 3a пункт 9). В commit message укажи `TASK-NNN [verify]: OK`. В dev_log entry — короткое «верифицирован TASK-XXX, регрессий нет».
6. **Если найдена регрессия / gap** — НЕ помечай verify-таск как done. Допиши follow-up таск в секцию `## 5. Follow-up tasks` в CONTEXT.md в формате:
   ```
   - [ ] **TASK-NNN-followup [impl]** — короткое название
     - Found by: TASK-XXX [verify]
     - Файлы: ...
     - Проблема: ...
     - Acceptance: ...
   - [ ] **TASK-NNN-followup-verify [verify]** — TASK-NNN-followup
   ```
   Verify-таск (текущий) пометь `[x]` ТОЛЬКО если решено что follow-up закроет проблему. Опиши юзеру в чат что нашёл и зачем follow-up.
7. Отчёт юзеру: что верифицировал, всё ли OK, есть ли follow-up.

### Шаг 4 — Завершение итерации

В чат юзеру выведи **компактный** отчёт:
- Какой таск взял (ID + title)
- Что сделал в одном-двух предложениях
- Текущий статус ledger'а: сколько `[x]` / `[~]` / `[ ]` всего, какая фаза в работе
- Был ли релиз и какая версия
- Какой следующий таск (ID + title) — чтобы юзер знал что закидать в следующую итерацию

Не делай больше одного таска за итерацию, даже если есть соблазн «прихватить мелочь рядом». Дисциплина важнее скорости — юзер хочет видеть постепенный прогресс.

### Запреты

- ❌ НЕ менять код вне scope текущего impl-таска
- ❌ НЕ релизить если фаза не закрылась полностью
- ❌ НЕ скипать verify-таски
- ❌ НЕ изобретать новые таски в CONTEXT (кроме секции follow-up из verify-сценария)
- ❌ НЕ создавать новые .md файлы в audit/ — только существующие
- ❌ НЕ пушить с `--no-verify`, `--force`, force-push в main, и не делать `git reset --hard` без явной команды юзера
- ❌ НЕ удалять devlog.txt, dist/, scripts/AppIcon* без явной просьбы юзера
- ❌ НЕ запускать `tccutil reset` (это дело юзера)
- ❌ НЕ трогать `.gitignore`, `Package.swift`, `setup_signing.sh` без явного указания

### Критерии «good enough» для одной итерации

- Один таск завершён (impl или verify) корректно
- Нет regression в build (`swift build` → 0 errors)
- ledger обновлён, статусы корректны
- Если был релиз — DMG'ы запакованы, тег запушен, GitHub Release создан, dev_log.json содержит phase-entry
- Юзер видит ясный отчёт: что сделано, что дальше
